                              Operator Loop v0.1
                              
  
                              
    scripts/trinity/useful_compute_operator_loop.py is
    a repeatable orchestrator for the full Useful Compute pipeline
    (task builder → worker → replay → governance
    → budget → proposal → payment draft). It loads
    each sibling script via importlib and writes a
    deterministic operator_run.json + a
    SHA256SUMS.txt manifest under the chosen
    --out-dir. Pass --resume at the same
    out-dir to skip already-completed steps; tampered artefacts
    abort the resume with rc=2.
  
                              
  
                                
    
                              • repeatable, deterministic with --pinned-time
                                • 
    
                              • resumable via --resume operator_run.json + sha256 verification
                                • 
    
                              • no wallet access: schema locks allow_wallet_access = const false
                                • 
    
                              • no broadcast authority: schema locks allow_broadcast = const false
                                • 
    
                              • no real signing: payment_draft always runs in unsigned-only or dry-sign mode
                                • 
    
                              • no subprocess: no sost-cli invocations from the loop
                                • 
    
                              • can start from an existing request.json (Sprint 5.22): pass --request-json <path> to skip the loop's task builder and import a manifest produced elsewhere — including a Sprint 5.21 --from-scientific-intake request
                                • 
  
                              
  
                              
    Trinity can re-run the pipeline; it cannot pay.
    The two human inflection points are unchanged from Sprint 5.17 /
    5.18: real signing requires a separate, manual call with its own
    token, and broadcasting requires another manual call with yet
    another token. The operator loop just removes the
    copy-paste-and-re-run friction between them.
  
                              

                              



                              
  
                              Scientific Prompt Intake v0.1
                              
  
                              
    scripts/trinity/scientific_prompt_intake.py is
    Trinity's local-only science inbox. It accepts a prompt plus
    any number of reference documents (.txt /
    .md / .json), hashes everything and
    writes one deterministic audit artifact
    TRINITY_SCIENTIFIC_PROMPT_INTAKE_<id>.json.
    Documents are recorded by basename + sha256 + size +
    a bounded preview; absolute paths and full content are
    deliberately NOT stored.
  
                              
  
                                
    
                              • local documents only — never reads from the network
                                • 
    
                              • hashes + previews — every input has a 64-hex sha256 + a 256-char (default) preview
                                • 
    
                              • no network — no HTTP / socket / WebSocket / FTP imports allowed at static-check time
                                • 
    
                              • no LLM call — no openai / anthropic / completion tokens; offline by design
                                • 
    
                              • deterministic — same prompt + documents + --pinned-time produce byte-identical artifacts
                                • 
    
                              • bounded — prompt ≤ 32 KiB, doc ≤ 1 MiB, ≤ 16 docs (all overridable downward)
                                • 
  
                              
  
                              
    Bridge to task generation lands in Sprint 5.21.
    The intake artifact becomes the input to
    useful_compute_task_builder --from-scientific-intake.
    v0.1 of the intake stops at the artifact: no task is decided,
    no LLM is invoked, no payment is initiated.

                              Autonomy Governor v0.1
                              
  
                              
    Constitutional layer that evaluates whether a proposed autonomous
    Trinity action would be permitted by the active policy. Ships in
    scripts/trinity/autonomy_governor.py with policy
    config/trinity_autonomy_governor.example.json.
  
                              
  
                                
    
                              • Observe only — v0.1 emits decision JSONs;
        it never executes the action it evaluates.
                                • 
    
                              • No network, no shell, no wallet, no broadcast.
        Static safety test fails CI if the governor source ever imports
        requests, subprocess, sign / broadcast
        primitives, or eval/exec.
                                • 
    
                              • Policy hash pinned at boot. Mutation of the
        policy file between load and decision blocks with
        policy_mutated_at_runtime.
                                • 
    
                              • Halt file is sacred. Presence of
        /etc/trinity/HALT blocks every action; Trinity must
        not have write permission on that path.
                                • 
    
                              • autonomous_sost_stocks hardcoded to 0.
        A non-zero value in the policy is rejected at load time. Enabling
        autonomous spending is a sprint upgrade, not a config edit.
                                • 
    
                              • Each decision cites the relevant threats from
        SECURITY.md in threat_refs (T01-T20).
        real_sign, broadcast_signed_transaction,
        wallet_access, constitution_change and
        register_new_source_tool always require a human.
                                • 
    
                              • External heartbeating lives in a future separate process,
        governor_watchdog.py, NOT in the governor itself —
        keeping the governor network-free is the whole point.
                                • 
  
                              
  
                              
    First integration into the operator loop landed in Sprint 5.24
    as an observe-only audit hook (see card below). v0.1 of the
    Governor itself remains a standalone evaluator with 44 tests

                              Sprint Release Runner
                              
  
                              
    Sprint 5.40 ships a single-shot
    preflight verifier that turns the release
    cut from a hand-typed sequence of git verbs into a
    reproducible report. It does NOT perform
    the release. The destructive verbs — push, merge, tag
    — remain in the operator's hands.
  
                              
  
                                
    
                              • What it does.
        scripts/trinity/sprint_release_runner.py verify
        … inspects the current branch, the HEAD
        commit, the base-ref commit, the tracked-dirty bit, the
        untracked-file count, the diff and commits-ahead vs the
        base, optionally runs pytest, and optionally
        walks a demo dir to surface Trinity artifacts identified
        by schema content, not filename. Output
        is a single
        trinity-sprint-release-report/v0.1 JSON plus
        a Markdown rendering with eight sections.
                                • 
    
                              • Read-only by construction. All
        subprocess calls use argv lists; the only
        git verbs allowed are seven read-only ones
        (rev-parse, status,
        diff, log, branch,
        ls-files, rev-list). The static
        safety test rejects the destructive argv literals
        "push", "merge",
        "tag", "reset",
        "checkout", "rm",
        "clean", "commit",
        "add", "stash"
        — the runner cannot mutate git state.
                                • 
    
                              • Artifact discovery by schema. The
        recursive demo-dir walk identifies each JSON by its
        schema field, so renamed files
        (daily-report.json,
        weird-name.json, …) are still
        recognised as
        autopilot_report / dashboard /
        daily_report /
        trial_pack_manifest and summarised
        (autopilot batches + safety_status, dashboard counts,
        daily report top_materials + cache_hits_total +
        workers_seen_total, trial_pack worker_id +
        expected_compute_output_sha256). Symlink-safe; capped at
        100 entries.
                                • 
    
                              • Hard gates.
        ready_to_release flips to false
        on branch mismatch, dirty tracked tree with
        --require-clean-tracked-tree, untracked
        files present without --allow-untracked,
        or any pytest failure / error / non-zero return code.
        Exit codes are 0 (ready),
        1 (not ready), 2 (setup error).
                                • 
    
                              • Seven const-true safety flags
        enforced at both script and schema level:
        no_git_push, no_git_merge,
        no_git_tag, no_wallet_access,
        no_signing, no_broadcast,
        no_network_required. The static lint also
        forbids requests / urllib /
        httpx / aiohttp /
        socket, the GitHub API surface
        (api.github.com, GITHUB_TOKEN,
        PyGithub), wallet / sign / broadcast tokens,
        and any LLM client.
                                • 
    
                              • Non-goals (documented). NOT a release
        tool. NOT a CI surrogate. NOT a daemon. NOT
        GitHub-aware (no API). NOT a wallet, signer, or
        broadcaster.
