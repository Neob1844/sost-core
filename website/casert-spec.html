Difficulty Specification

cASERT — Contextual ASERT
Difficulty Adjustment

A consensus-critical stability overlay that extends ASERT with anti-stall decay and multi-level difficulty hardening for ConvergenceX Proof-of-Work.

CONSENSUS | Integer-only arithmetic · Q16.16 fixed-point · Per-block retargeting

ASERT Base Inspiration

Origin & Core Formula
BCH 2020

ASERT (Absolutely Scheduled Exponentially-weighted Rising Targets) was designed by Mark Lundeberg and Jonathan Toomim for Bitcoin Cash (November 2020 hard fork, aserti3-2d specification). It replaced the older DAA with a mathematically clean exponential moving average that retargets every block. cASERT extends their work with anti-stall decay and level-based hardening for ConvergenceX.

Key principle: ASERT is anchor-based. The difficulty for any block can be computed directly from the anchor block and the elapsed time, without iterating through intermediate blocks. This makes it stateless and resistant to manipulation of intermediate timestamps.

How cASERT Extends ASERT for ConvergenceX

ConvergenceX PoW has a stability_metric (gradient descent convergence) that can be tightened. cASERT monitors how far ahead of schedule the chain is and introduces discrete hardening levels to prevent fast-block attacks.

UNIDIRECTIONAL

Hardens Only

cASERT only HARDENS difficulty, never relaxes it. When the chain is behind schedule, ASERT alone handles the adjustment. This prevents attackers from exploiting difficulty drops.

SCALE = LEVEL

Proportional Response

The hardening scale equals the level number. Higher levels impose stricter stability_metric thresholds on ConvergenceX, making it progressively harder to produce fast blocks.

ANTI-STALL

Decay After 7200s

If no block is found for 7,200 seconds (2 hours), a decay_factor gradually reduces the cASERT scale. This prevents mining deadlock when hashrate drops suddenly.

Anti-Stall Mechanism
DECAY PROTOCOL

The anti-stall decay ensures liveness under sudden hashrate loss. If no valid block arrives for DECAY_START seconds, the cASERT hardening is gradually reduced. This decay is deterministic and computed from the timestamp of the previous block, making it fully verifiable by all nodes.

ConditionResponse
Block interval < 7,200sNo decay — cASERT level applies at full scale
Block interval >= 7,200sdecay_factor reduces cASERT scale progressively
ResultMining continues even if hashrate drops dramatically
Design rationale: Discrete levels (rather than a continuous function) eliminate consensus ambiguity from rounding differences between implementations. Every conforming node will compute exactly the same level for the same chain state.
