[size=14pt][b]ConvergenceX v2.0 — Technical Summary[/b][/size]

ConvergenceX is not a simple hash lottery. Each mining attempt requires solving a 32×32 linear system through 100,000 sequential rounds of integer-only gradient descent, then proving the solution sits in a stable basin of attraction.

[list]
[li][b]Memory-hard:[/b] 4 GB dataset + 4 GB scratchpad = ~8 GB total mining memory. Node validation ~500 MB.[/li]
[li][b]Per-block program:[/b] 256-operation program changes every block.[/li]
[li][b]Difficulty:[/b] avg288 + median288 controller plus ConvergenceX equalizer profiles.[/li]
[li][b]Integer-only consensus:[/b] zero floating-point operations, Q16.16 fixed-point difficulty encoding.[/li]
[/list]

[size=14pt][b]cASERT Calibration Status — V10 Granular Relief at Block 6700[/b][/size]

A mandatory calibration of the staged-relief cascade activates at [b]block 6700[/b]. The change refines [b]how[/b] the relief descends; it does not change [b]who[/b] is allowed to mine, [b]how much[/b] they earn, or [b]how[/b] blocks are validated.

[code]
V9 (current):  drop 3 levels every 60 s, starting at 540 s, floor E7
V10 (≥ 6700):  drop 1 level every 60 s, starting at 600 s, floor E7
[/code]

[b]No supply change. No reward change. No wallet action. No GPU ban. No pool ban. Hashrate-proportional competition preserved.[/b]

Anti-stall, lag clamp, future-timestamp drift cap (+60 s) and bitsQ are all unchanged. Monte Carlo fairness simulation (5,000 blocks per scheme, seed 42, 5-miner mix sized to the live chain) shows every miner stays within ±0.6 % of its hashrate share — on par with V9 — while the count of E7-class blocks drops ~22 % (1,475 vs 1,896 / 1,811), so each block lands at a healthier intermediate profile (H6 / H4 / H1 / B0 / E1) and the bitsQ controller no longer needs to push back the other way.

[code]
cd ~/SOST/sostcore/sost-core && git pull --ff-only origin main

  cmake --build . --target sost-node sost-miner -j$(nproc)

Make sure NTP is active ([code]sudo timedatectl set-ntp true[/code]) — the +60 s future-drift cap from the 6550 fork is unchanged.

[size=14pt][b]Core Economic Design[/b][/size]

Every SOST block automatically allocates:

[code]
50% → miner
25% → Gold Funding Vault
25% → PoPC Pool
[/code]

This is a consensus rule. A block with the wrong split is invalid.

The Gold Vault is designed to accumulate protocol-level gold reserves over time. The PoPC Pool powers Proof of Personal Custody, a future mechanism intended to reward holders who self-custody gold and prove it.

SOST is [b]not pegged to gold[/b]. Holders have no redemption right against the reserve. There is no stablecoin mechanic, no oracle dependency for consensus and no algorithmic peg.

[size=14pt][b]How to Mine[/b][/size]

[b]Requirements:[/b] Linux x86_64, 8 GB RAM minimum, CPU-only.

[code]
git clone https://github.com/Neob1844/sost-core.git
cd sost-core && mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
cp ../genesis_block.json .

# Direct P2P sync
./sost-node --genesis genesis_block.json \
  --connect seed.sostcore.com:19333 \
  --rpc-user YOUR_USER --rpc-pass YOUR_PASS \
  --profile mainnet --p2p-enc off

# Bootstrap fast-start
curl -o chain.json https://sostcore.com/bootstrap-chain.json

./sost-node --genesis genesis_block.json --chain chain.json \
  --connect seed.sostcore.com:19333 \
  --rpc-user YOUR_USER --rpc-pass YOUR_PASS \
  --profile mainnet --p2p-enc off

# Verify height
curl -s --user YOUR_USER:YOUR_PASS \
  -d '{"method":"getblockcount","params":[],"id":1}' \
  http://127.0.0.1:18232

# Run miner after node is synced
./sost-miner --address YOUR_SOST_ADDRESS \
  --rpc 127.0.0.1:18232 \
  --rpc-user YOUR_USER --rpc-pass YOUR_PASS \
  --blocks 999999 --profile mainnet
[/code]
